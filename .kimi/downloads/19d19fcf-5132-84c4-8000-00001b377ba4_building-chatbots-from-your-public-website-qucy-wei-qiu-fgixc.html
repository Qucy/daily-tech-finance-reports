Building Chatbots from Your Public Website

In this article, we’ll explore how to build a chatbot that uses your public website content as its knowledge base, leveraging open-source libraries and simple architectural patterns.

1. Crawling and Converting Website Content

The first step is to crawl your website and normalize the content so it can be efficiently retrieved and understood by a large language model (LLM). While libraries like BeautifulSoup, Crawlee or Scrapy can scrape HTML, they often leave you with fragmented or messy markup that requires a lot of custom post-processing.

I discovered Crawl4AI, which provides built-in rules to transform raw HTML into clean Markdown—a format much easier for LLMs to parse. For example, with just a few lines of code you can scrape every page of hsbc.com.hk, and group each subdomain into its own folder.

And convert each HTML page into a Markdown file, preserving image URLs and links for later reference.

2 - Indexing and Retrieval: Introducing “Sitemap RAG”

Traditional vector-based retrieval can miss important contextual groupings—e.g., if a user asks, “What’s the best credit card for online shopping?”, you need every card’s details to compare them fairly. One option is Graph-RAG, but building and maintaining a complete knowledge graph can be complex and brittle.

Instead, we propose Sitemap RAG—leveraging the fact that most public websites are already organized into logical subdomains and sections. By extending the existing sitemap into a tree structure, we can associate each node with its scraped content (and attachments, if any):

We then load this tree into a relational database with three tables:

  1. sites: Basic site metadata (e.g., domain, crawl date).

  2. site_nodes: The tree hierarchy (parent-child relationships).

  3. content: The raw Markdown for each node, indexed by URL/path.

3. Retrieval Workflow

When a user asks a question, we follow a simple, guided retrieval process:

  1. Level-1 Selection Pass the top-level sitemap (e.g., “Accounts,” “Credit Cards,” etc.) to the LLM and let it select top 3 subsite maps.

  2. Subnode Selection For each selected category, present its subtree to the LLM and ask it to pick the most relevant nodes and URL paths.

  3. Content Assembly Retrieve all Markdown content for the chosen paths and concatenate them into the final context. The LLM can then compare details across pages (e.g., multiple credit-card offers) and generate a comprehensive response.

4. Handling Dynamic Components

Many websites include interactive elements—mortgage calculators, loan estimators, etc. To support these, you can:

  1. Predefine Component Templates Write small React or Vue widgets for each calculator or interactive element.

  2. Intent Detection When the user’s query indicates they want to use a calculator (e.g., “What would my monthly repayment be on a personal loan?”), embed the corresponding widget directly in the chatbot interface.

5. Why This Matters

By combining clean Markdown conversion, a sitemap-driven RAG index, and dynamic component embedding, you can deliver a chatbot experience that feels thoroughly modern:

  • Multimodal: Includes images and interactive widgets.

  • Comprehensive: Retrieves and compares all relevant pages.

  • Maintainable: Relies on your existing sitemap structure rather than a bespoke knowledge graph.

It isn’t trivial to set up—but the payoff is a chatbot that truly feels like a 2025 experience, not a static FAQ bot from 2010.
